## ✅ 안건 2 — 거래처 등록 & 업종 체계 (확정)

**구조**: 2단 업종 + 기타 수기 + (향후) 색상 태그
- 업종 **대분류**(필수) 선택 → "의료" 선택 시 **진료과목**(필수) 하위 선택
- 대분류/진료과 모두 **"기타" 선택 시 수기 입력칸** 노출
- 업종별 **색상 태그** → 거래처 목록·캘린더·카드에서 자동 표시(향후 활용)

**데이터 방식 (확정)**: **관리자 마스터 방식**
- 업종 대분류·진료과목을 **설정 화면에서 관리자가 추가/수정·색상 지정** (코드 수정 불필요)
- 마스터 테이블 예: `IndustryCategory`(name, parentId, colorTag, isActive, sortOrder)
  - 대분류는 parentId=null, 진료과목은 대분류를 parent로
- `Client`에 `industryCategoryId`(FK) + 기타용 `industryCustom`(String?) 추가

**초기 시드(편집 가능)**
- 대분류: 의료 / 뷰티(피부·에스테틱·네일) / 요식(식당·카페) / 법률 / 교육(학원) / 부동산 / 쇼핑몰·커머스 / 기타
- 진료과목: 피부과 · 성형외과 · 정형외과 · 치과 · 한의원 · 안과 · 비뇨의학과 · 산부인과 · 이비인후과 · 내과 · 정신건강의학과 · 재활의학과 · 가정의학과 · 통증의학과 · 기타

> ⚠️ 마이그레이션 필요: `IndustryCategory` 신규 모델 + `Client.industryCategoryId/industryCustom` 컬럼 추가.

## ✅ 안건 3 — 거래처별 업무 카테고리 (확정)

**방식**: 업무 카테고리도 **마스터 테이블**(`WorkCategoryMaster`: name, group, colorTag, isLocked, isActive, sortOrder).
기존 `WorkItem.category`(enum)는 마스터 FK 또는 코드값 참조로 전환.

**거버넌스 (마스터 공통 규칙 — 업종·업무카테고리 모두 적용)**
- **ADMIN**: 마스터 항목 추가/수정/삭제 가능 (잠기지 않은 항목).
- **SUPER_ADMIN**: 추가로 **잠금(lock)** 권한. 잠긴 항목은 ADMIN이 수정/삭제 불가, 최고관리자만 관리.
- 잠금은 핵심 카테고리(예: 브랜드블로그, 플레이스 등)의 실수 변경 방지용.

**그룹 + 13개 시드 (편집 가능)**

| 그룹 | 카테고리 |
|---|---|
| 블로그 | 브랜드블로그 작성 · 블로그 배포 · 블로그 상위노출 |
| 검색광고 | 파워링크(네이버 검색광고) 🆕 |
| 지도 | 플레이스 순위관리 |
| SNS | SNS 관리/배포 |
| 웹 | SEO(웹사이트·테크니컬) 🆕 · GEO(생성형엔진/AI검색) 🆕 · 홈페이지 관리 🆕 |
| 운영 | 계정관리 · 영수증 리뷰 · 월간 보고서 작성 · 방문자수/성과 집계 |

> 거래처별로 "어떤 그룹의 서비스를 계약했는지"를 한눈에 볼 수 있게 그룹 단위 표시.
> ⚠️ 마이그레이션: `WorkCategoryMaster` 신규 + `WorkItem.category` 연결 방식 조정.

## ✅ 안건 4 — 거래처 채널계정 구조 (확정)

**채널 종류 (마스터 방식, 시드 — 편집 가능)**
네이버 블로그 · 인스타그램 · 페이스북 · 유튜브 · 스레드 · 네이버 플레이스 · 홈페이지 · 네이버 검색광고(파워링크) · 네이버 애널리틱스/GA · 카카오채널 · 기타
- 채널별 저장: 채널명, URL/주소, 계정 아이디, 담당자, 활성여부, 대표채널 여부.

**로그인 정보 = 전체 암호화 저장 (확정)**
- 아이디 + **비밀번호 암호화 저장**(평문 금지). AES-256-GCM 등 대칭암호 + 키는 환경변수(`CREDENTIAL_ENC_KEY`), 절대 로그/응답에 평문 노출 금지.
- **열람·관리 최고 권한 = 최고관리자(SUPER_ADMIN)**.
- 모든 자격증명 **열람을 AuditLog에 기록**.
- ✅ 담당자(마케터)는 **본인 배정 거래처 계정만 열람 허용**(권장안 채택). 최고관리자는 전체. 모든 열람은 감사기록.

**직원 가입/인증 = 이메일 확인(인증) 방식 (확정 — 기존 Kakao 방향 대체)**
- 이메일 인증으로 가입. 최고 관리 권한은 최고관리자.
- ⚠️ 기존 "Kakao OAuth 유지"(plan 문서) 결정을 **이메일 인증 가입으로 변경**. Kakao 병행 여부는 추후 확인.

> ⚠️ 마이그레이션/보안: `ClientAccount`에 암호화 자격증명 필드 추가, 채널 마스터 테이블, 자격증명 열람 감사. 인증 방식 변경(이메일 인증)에 따른 Auth 설정 조정.

## ✅ 추적/감사 강화 — 로그인·활동 이력 (확정, 전 영역 공통)

최고관리자가 "누가 언제 무엇을 했는지" 추적 가능하게:

1. **로그인 이력 (`LoginHistory` 신규)**: userId, 일시, IP, UserAgent, 성공/실패, 로그아웃 시각. (이상 로그인 감지 토대)
2. **활동 이력 (기존 `AuditLog` 확장)**: 쓰기뿐 아니라 **민감 열람**(거래처 자격증명 조회 등)·중요 조회까지 기록. before/after·actor·IP·UA 유지.
3. **추적 화면 (최고관리자 전용, 설정/베놈단 내)**: 사용자별·기간별·대상별 필터 + 검색. 로그인 이력 + 활동 이력 통합 뷰.
4. 보존: 일정 기간(예: 1년) 보관 정책(추후 확정).

> 이로써 자격증명 암호화 저장 + 담당자 열람 허용의 **위험을 감사추적으로 상쇄**.
> ⚠️ 마이그레이션: `LoginHistory` 신규, AuditLog 활용 확대, 추적 UI.

## ✅ 안건 5 — 입금현황 & 결제 (확정)

**결제 방식 실태**: 계좌이체 95% / 카드 5%. → **이체 중심 설계**.
**자동화 목표**: **반자동(레벨 2)** — 내역 불러와 버튼 매칭(대사).

**단계별 구현**
1. **1차(수기 기반)**: 청구서(BillingRecord) 생성 → 수기 입금기록(PaymentRecord) → 상태(미입금/부분입금/입금완료/연체) 자동 계산. 합계 규칙은 V1에 이미 검증됨.
2. **반자동 대사(입금 스프린트)**: **은행 거래내역 업로드(CSV)** → 금액·거래처명으로 **후보 자동매칭** → 담당/경리가 **버튼으로 확정** → 청구건 상태 자동 변경 + Audit.
3. **카드 5%**: 소량이라 1차는 수기. PG(토스/카카오페이) webhook 자동화는 비중 낮아 후순위(필요 시 V3).

**V3(후순위, 사업자 인증·제휴·유료 필요)**: 오픈뱅킹/펌뱅킹 **실시간 계좌 자동조회**, 세금계산서 자동발행.

> 스키마: BillingRecord/PaymentRecord 재사용. 반자동용 `BankTransaction`(임포트 내역) 신규 + 매칭 상태. PaymentProvider enum(KAKAOPAY/TOSSPAYMENTS/INICIS) 기존 활용.
