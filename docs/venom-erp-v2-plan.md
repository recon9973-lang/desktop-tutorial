# 베놈 ERP V2 기획서 (브레인스토밍 / 초안)

작성일: 2026-06-30
대상 코드: `recon9973-lang/marketing-agency-erp` (브랜치 `erp-v1`, Next.js App Router + Prisma + PostgreSQL + Auth.js/Kakao)
회사: 주식회사 베놈 — 병원마케팅 대행사 (대구, 대표 김보형, 1661-4142)
문서 성격: V1 핸드오프 + 실제 스키마 검토 기반 V2 기획 초안. **결정 대기 항목은 ❓로 표시.**

---

## 0. 한 줄 요약

V1은 "읽기는 되는데 쓰기가 없는 ERP"다. 데이터 모델·권한·도메인 규칙·테스트는 검증돼 있고, **V2의 본질은 새 모델 추가가 아니라 쓰기(server action) + 폼 + 상태전이 검증 + 감사로그 연결**이다. 스키마에 이미 상태 enum과 연동용 필드가 다 있어 작업 난도는 낮은 편이다.

---

## 1. 현재 상태 진단 (실제 코드 확인 결과)

### 1.1 잘 되어 있는 것
- 역할 3종(`SUPER_ADMIN` / `ADMIN` / `MARKETER`) + `AccessScope`(adminId, marketerId, clientId, allMarketers, allClients) 기반 범위 조회.
- 거래처 담당자 배정(`Client.assignedMarketerId`).
- 9개 화면의 읽기 전용 대시보드(대시보드/거래처/업무/캘린더/휴가/정산·지출/보고서/설정).
- 도메인 규칙(업무 상태전이, 휴가 잔여일, 청구/미수/지출 합계) + 테스트 42개 통과, 빌드·e2e 통과.

### 1.2 비어 있는 것 (= 운영 불가 사유, V2 핵심)
거의 모든 **쓰기 액션**:
- 거래처 등록/수정/비활성화, 담당자 배정 변경
- 업무 생성/상태변경/진행메모 저장/첨부
- 휴가 신청/승인/반려, 캘린더 자동 반영
- 청구서 생성, 입금 기록, 지출 등록/검토
- 보고서 작성/성과지표 입력/검토/전달, 첨부
- `AuditLog` 실제 DB 저장 (헬퍼만 존재)
- 직원 초대/역할 변경/AccessScope 관리 UI
- 운영 관리자 초기 계정 생성 플로우, 파일 업로드 인프라

### 1.3 데이터 모델 (이미 V2를 감당할 수준)
주요 모델 16종 + enum 16종 확인. 상태 enum이 촘촘해 추가 모델이 거의 불필요:

| 모델 | V2에서 할 일 |
|---|---|
| `User` / `AccessScope` | 초대·역할변경·범위관리 **쓰기 액션** |
| `Client` / `ClientAccount` | 등록·수정·배정 폼 + 상세페이지 |
| `WorkItem` / `WorkTemplate` | 생성·상태전이·메모 + 반복 템플릿 자동생성 |
| `LeaveRequest` / `LeavePolicy` | 신청·승인·반려 + 잔여일 차감 + 캘린더 반영 |
| `BillingRecord` / `PaymentRecord` | 청구 생성·입금 기록 (PG필드는 이미 존재→V3) |
| `ExpenseRecord` / `FinancialAccount` | 지출 등록·검토 (sync필드 이미 존재→V3) |
| `Report` | 작성·지표(JSON)입력·검토·전달 |
| `CalendarEvent` | 내부 일정 생성 + 타 모델 자동 이벤트 (external sync→V3) |
| `AuditLog` | 모든 상태변경 시 저장 (beforeState/afterState 필드 이미 존재) |

## 2. 작업 환경 결정 (선행 필수) ❓

핸드오프가 가리키는 ERP 코드는 `recon9973-lang/marketing-agency-erp`(별도 레포)다. 현재 작업 세션은 `desktop-tutorial`(홈페이지) 레포로만 권한이 한정돼 있어 ERP 레포 clone/push가 막힌다.

**선택지:**
- **(A) ERP 레포를 작업 세션 권한에 추가** — 가장 깔끔. 기존 erp-v1 위에서 V2를 바로 구현/푸시.
- **(B) ERP 레포로 스코프된 새 세션에서 작업** — 분리 운영, 홈페이지와 무관.
- (C) 기획 문서만 이 레포에 보관(현재 이 문서) — 코드 착수는 보류.

> 권장: **(A) 또는 (B)**. ERP는 홈페이지와 분리된 서브도메인 앱이므로, 장기적으로 ERP 레포에서 독립 작업하는 (B)가 깔끔하다.

## 3. 운영 우선순위 (사용 빈도 기반) ❓

매일/주/월 사용 빈도와 의존관계로 본 우선순위:

| 순위 | 영역 | 빈도 | 이유 |
|---|---|---|---|
| 1 | **업무관리 CRUD + 상태전이 + 진행메모** | 매일(마케터) | 가장 자주 쓰는 화면. ERP의 심장 |
| 2 | **거래처 등록/배정/수정** | 수시(관리자) | 업무·정산·보고서가 모두 거래처에 의존 |
| 3 | **휴가 신청/승인/반려** | 주 단위 | 플로우 단순, 빨리 완성 가능 |
| 4 | **청구/입금/지출 입력** | 월 단위 | 정산 마감. 합계 규칙은 이미 검증됨 |
| 5 | **보고서 작성/검토/전달** | 월 단위 | 거래처 성과 산출물 |
| 6 | **AuditLog 실저장 + 설정(직원/권한)** | 상시(배경) | 책임추적·운영 거버넌스 |

> 권장 1차 스프린트: **1(업무) + 2(거래처)**. 이 둘이 "매일 쓰는 화면"이고 나머지의 데이터 토대다.

## 4. 병원마케팅 특화 (베놈 차별화) ❓

베놈은 일반 대행사가 아니라 **병원마케팅** 대행사. 일반 ERP 골격 위에 얹을 후보:

| 후보 | 효과 | 비용 | 비고 |
|---|---|---|---|
| **네이버 순위/방문자수 자동수집** | 높음 | 중 | 홈페이지 레포에 네이버 키워드 API 프록시(`api/kw-proxy.js`)가 이미 있음 → 재활용. `Report.metrics`(JSON)에 적재. V1은 V2로 미뤘으나 **조기 도입 가치 큼** |
| **의료광고 심의(의료법) 체크리스트** | 중 | 낮음 | 콘텐츠 발행 전 의료법 위반 소지 점검. `WorkItem`에 체크리스트 필드/연결 |
| **플레이스/키워드 순위 추적 화면** | 중 | 중 | 병원별 진료과·지역 키워드, 플레이스 순위 모니터링 |
| 특화 없이 일반 ERP 먼저 | — | — | 표준 ERP 완성 후 특화 |

> 권장: **네이버 자동수집을 조기 도입**(인프라 보유 + 차별화), **심의 체크리스트는 가볍게 동반**. 순위추적 화면은 그 다음.

## 6. 공통 설계 원칙 (V2 구현 시 지킬 것)

1. **모든 쓰기는 server action + Zod 검증 + 권한 가드(AccessScope) 통과 후 실행.** 읽기 권한 모델을 쓰기에도 동일 적용.
2. **상태 변경은 도메인 규칙 함수를 거친다.** 이미 있는 업무/휴가 상태전이 규칙을 쓰기 경로에 연결(불법 전이 차단).
3. **상태 변경 = AuditLog 1건.** before/after 스냅샷 저장을 액션 공통 래퍼로.
4. **타 모델 자동 캘린더 이벤트.** 업무 마감·휴가 승인·보고서 마감 시 `CalendarEvent` 자동 생성/갱신.
5. **낙관적 UI 대신 명확한 성공/실패 토스트 + 서버 재검증.** 운영 데이터 정합성 우선.
6. **초기 데이터 import(엑셀/CSV).** 거래처·직원·담당배정 초기 적재용 → 도입 마찰 최소화.

## 7. 배포·운영 체크리스트 (V1 핸드오프 + 보강)

- [ ] 운영 도메인 확정 (권장: `erp.<베놈도메인>` 서브도메인, 홈페이지와 분리)
- [ ] 운영 PostgreSQL + 백업 정책
- [ ] 환경변수: `DATABASE_URL`, `AUTH_SECRET`(강력 난수), `AUTH_URL`, `AUTH_KAKAO_ID/SECRET`

- [ ] Kakao Developers 앱 + Redirect URI: `https://erp.<도메인>/api/auth/callback/kakao`
- [ ] 초기 SUPER_ADMIN 계정 seed/수동 등록
- [ ] 직원 역할·관리자 AccessScope를 실제 조직도와 일치
- [ ] 거래처별 담당자 정확 배정
- [ ] 책임자 지정: 청구/입금 관리자, 지출 검토권자, 연차 승인권자, 보고서 검토·전달자
- [ ] 개인정보/계정정보 보관·마스킹 정책

## 8. 결정이 필요한 항목 (요약) ❓

1. **작업 환경**(§2): ERP 레포 권한 추가(A) / 새 세션(B) / 기획만(C)?
2. **1차 스프린트 범위**(§3): 업무+거래처 권장 — 동의 여부?
3. **병원 특화**(§4): 네이버 자동수집 조기 도입 + 심의 체크 권장 — 동의 여부?
4. **인증 방식**: Kakao OAuth 유지 vs 이메일+초대(소규모면 운영 단순) ?
5. **초기 데이터 import** 필요 여부 및 양식(엑셀/CSV)?

위 5개가 정해지면 1차 스프린트의 상세 작업 분해(WBS)와 server action 목록까지 바로 작성한다.
