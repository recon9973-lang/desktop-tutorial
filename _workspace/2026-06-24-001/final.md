# 효과적인 에이전트 만들기

*2024년 12월 19일 게재*

지금까지 수십 개 팀과 함께 LLM 에이전트를 만들어봤다. 거의 예외 없이, 잘 돌아가는 구현에는 공통점이 있었다. 복잡한 프레임워크나 전용 라이브러리를 쓰는 게 아니라, 단순하고 조합하기 쉬운 패턴을 쓴다는 것.

고객과 일하며 쌓은 경험, 그리고 우리가 직접 에이전트를 만들면서 배운 것들을 정리했다. 실제로 에이전트를 짜는 개발자에게 실용적인 조언이 됐으면 한다.

---

## 에이전트란 무엇인가

"에이전트"를 어떻게 정의하느냐는 사람마다 다르다. 어떤 팀은 다양한 도구를 써가며 장시간 혼자 돌아가는 완전 자율 시스템을 에이전트라 부른다. 어떤 팀은 미리 정해진 흐름을 따르는 구현에도 그 이름을 붙인다. Anthropic에서는 이 모두를 "에이전틱 시스템(agentic system)"이라 통칭하면서, 아키텍처 측면에서 두 가지를 구분한다.

- **워크플로우** — LLM과 도구를 미리 짜둔 코드 경로로 조율하는 시스템
- **에이전트** — LLM이 스스로 프로세스와 도구 사용을 결정하며 어떻게 목표를 달성할지 직접 제어하는 시스템

아래에서 둘 다 자세히 다룬다. 부록 1("현장의 에이전트")에서는 고객들이 실제로 이런 시스템에서 가치를 찾은 두 가지 분야를 소개한다.

---

## 언제 에이전트를 쓸 것인가

LLM으로 뭔가를 만들 때는 가능한 한 단순한 해법부터 찾고, 꼭 필요할 때만 복잡도를 올리는 게 맞다. 에이전틱 시스템을 아예 안 만드는 게 나을 때도 많다. 에이전틱 시스템은 대개 지연 시간과 비용을 더 쓰는 대신 성능을 높이는 트레이드오프다. 그 트레이드오프가 언제 말이 되는지 먼저 따져봐야 한다.

복잡도를 올릴 이유가 생겼다면, 잘 정의된 작업엔 워크플로우가 예측 가능성과 일관성을 준다. 유연성과 모델이 스스로 판단하는 능력이 필요한 규모의 작업엔 에이전트가 맞다. 다만 많은 경우, 단일 LLM 호출에 검색(retrieval)과 인컨텍스트 예시만 잘 붙여도 충분하다.

---

## 프레임워크를 쓸 것인가

에이전틱 시스템 구현을 도와주는 프레임워크가 많다.

- Claude Agent SDK
- AWS의 Strands Agents SDK
- Rivet — 드래그 앤 드롭 방식의 LLM 워크플로우 빌더
- Vellum — 복잡한 워크플로우를 만들고 테스트하는 GUI 도구

이런 프레임워크는 LLM 호출, 도구 정의와 파싱, 호출 체이닝 같은 저수준 작업을 단순화해 빠르게 시작하게 해준다. 그런데 추상화 계층이 하나씩 끼면서 아래 프롬프트와 응답이 잘 안 보인다. 디버그가 어려워지고, 단순하게 가도 될 것을 괜히 복잡하게 만들고 싶어지기도 한다.

우리가 권하는 방법은 LLM API를 직접 쓰는 것이다. 대부분의 패턴은 코드 몇 줄로 구현할 수 있다. 프레임워크를 쓴다면, 그 안에서 무슨 일이 일어나는지는 꼭 파악해둬야 한다. "내부가 이럴 거야"라는 잘못된 가정이 고객 오류의 흔한 원인이다.

샘플 구현은 쿡북을 참고하라.

## 빌딩 블록, 워크플로우, 에이전트

에이전틱 시스템에서 실제로 자주 보이는 패턴들을 살펴보자. 기본 단위인 확장된 LLM부터 시작해서, 단순한 조합 워크플로우를 거쳐 자율 에이전트까지 복잡도를 높여가며 설명한다.

### 기본 단위: 확장된 LLM

에이전틱 시스템의 기본 블록은 검색, 도구, 메모리로 보강된 LLM이다. 지금의 모델은 이런 기능을 능동적으로 활용할 수 있다. 검색 쿼리를 스스로 만들고, 적절한 도구를 고르고, 뭘 기억해둘지 판단한다.

구현에서 두 가지에 집중하길 권한다. 하나는 내 사용 사례에 맞게 이 기능들을 다듬는 것, 다른 하나는 LLM이 쓰기 쉽고 문서화가 잘 된 인터페이스를 제공하는 것이다. 구현 방법은 여러 가지가 있는데, 최근 공개한 Model Context Protocol(MCP)을 쓰면 간단한 클라이언트 구현으로 써드파티 도구 생태계와 연결할 수 있다.

이하 본문에서는 각 LLM 호출이 이런 보강 기능을 갖추고 있다고 가정한다.

---

### 워크플로우 ① 프롬프트 체이닝

프롬프트 체이닝은 하나의 작업을 여러 단계로 쪼개서, 각 LLM 호출이 이전 단계의 출력을 받아 처리하는 방식이다. 중간 단계마다 프로그래밍 방식으로 검사(아래 다이어그램의 "게이트")를 끼워 흐름이 제대로 가고 있는지 확인할 수 있다.

**언제 쓰나:** 작업을 고정된 하위 작업으로 깔끔하게 쪼갤 수 있을 때 적합하다. 각 LLM 호출을 더 쉬운 작업으로 만들어 지연 시간을 더 써도 정확도를 높이는 게 핵심 목표다.

**예시:**
- 마케팅 카피를 쓰고, 그걸 다른 언어로 번역한다
- 문서 개요를 짜고, 특정 기준을 충족하는지 확인한 뒤, 개요를 바탕으로 본문을 쓴다

---

### 워크플로우 ② 라우팅

라우팅은 입력을 분류해서 적합한 후속 작업으로 보내는 방식이다. 관심사를 분리하고, 더 특화된 프롬프트를 쓸 수 있다. 이 워크플로우 없이 특정 입력에 최적화하면 다른 입력 성능이 떨어지는 경우가 많다.

**언제 쓰나:** 별도로 처리하는 게 나은 카테고리가 뚜렷이 구분되고, 분류 자체를 LLM이나 전통적인 분류 모델·알고리즘으로 정확하게 할 수 있는 복잡한 작업에 잘 맞는다.

**예시:**
- 고객 문의를 종류별로 분류해 각각 다른 프로세스, 프롬프트, 도구로 보낸다 (일반 질문, 환불 요청, 기술 지원 등)
- 쉽거나 자주 나오는 질문은 Claude Haiku 4.5 같은 가볍고 저렴한 모델로, 어렵거나 특이한 질문은 Claude Sonnet 4.5 같은 더 강력한 모델로 보내 성능과 비용을 맞춘다

---

### 워크플로우 ③ 병렬화

LLM이 작업을 동시에 처리하고 그 출력을 코드로 집계하는 방식이다. 두 가지 형태가 있다.

- **섹셔닝(Sectioning):** 작업을 독립적인 하위 작업으로 나눠 병렬 실행
- **투표(Voting):** 같은 작업을 여러 번 돌려 다양한 출력을 얻음

**언제 쓰나:** 나눈 하위 작업을 속도를 위해 병렬 처리할 수 있을 때, 또는 더 높은 신뢰도를 위해 여러 관점이나 시도가 필요할 때 효과적이다. 고려할 게 많은 복잡한 작업에서는 각 고려 사항을 별도 LLM 호출로 처리하면 집중도가 높아져 대체로 성능이 좋아진다.

**예시:**

*섹셔닝:*
- 하나의 모델 인스턴스는 사용자 쿼리를 처리하고, 다른 인스턴스는 부적절한 내용을 걸러내는 가드레일을 구현한다. 같은 LLM 호출 하나로 가드레일과 핵심 응답을 같이 처리하는 것보다 성능이 낫다.
- LLM 성능 평가를 자동화할 때 각 LLM 호출이 서로 다른 측면을 평가하도록 한다.

*투표:*
- 코드의 취약점을 검토할 때 여러 프롬프트가 각자 코드를 검토하고 문제를 발견하면 플래그를 단다.
- 특정 콘텐츠가 부적절한지 판단할 때 여러 프롬프트가 서로 다른 측면을 평가하거나 다른 투표 임계값을 적용해 오탐/미탐을 조율한다.

---

### 워크플로우 ④ 오케스트레이터-워커

중앙의 LLM이 작업을 동적으로 나누고, 워커 LLM에 위임하고, 결과를 종합한다.

**언제 쓰나:** 어떤 하위 작업이 필요한지 미리 예측하기 어려운 복잡한 작업에 잘 맞는다. 예를 들어 코딩 작업에서는 바꿔야 할 파일 수나 각 파일의 변경 성격이 작업에 따라 달라진다. 병렬화와 구조가 비슷해 보이지만 핵심 차이는 유연성이다. 하위 작업이 미리 정해지지 않고, 오케스트레이터가 입력에 따라 그때그때 결정한다.

**예시:**
- 매번 여러 파일을 복잡하게 수정해야 하는 코딩 제품
- 가능성 있는 정보를 여러 출처에서 모아 분석해야 하는 검색 작업

---

### 워크플로우 ⑤ 평가자-최적화자

하나의 LLM 호출이 응답을 생성하고, 다른 LLM이 평가와 피드백을 루프로 제공한다.

**언제 쓰나:** 명확한 평가 기준이 있고 반복적인 개선이 측정 가능한 가치를 낼 때 특히 효과적이다. 두 가지 징표가 있다. 사람이 피드백을 말로 표현하면 LLM 응답이 눈에 띄게 나아지는 경우, 그리고 LLM 스스로 그런 피드백을 줄 수 있는 경우다. 사람 작가가 퇴고를 반복하며 완성도를 높이는 과정과 비슷하다.

**예시:**
- 번역가 LLM이 처음엔 잡아내지 못한 뉘앙스를 평가자 LLM이 짚어주는 문학 번역
- 포괄적인 정보를 모으기 위해 검색과 분석을 여러 번 반복해야 하는 복잡한 검색 작업 (평가자가 추가 검색이 필요한지 판단)

## 에이전트

LLM이 복잡한 입력 이해, 추론과 계획, 도구의 안정적 사용, 오류 복구 같은 핵심 역량을 갖춰가면서 에이전트가 실제 서비스에 등장하고 있다. 에이전트는 사람의 지시나 대화로 작업을 시작한다. 작업이 명확해지면 스스로 계획하고 움직이되, 추가 정보나 판단이 필요할 때 다시 사람에게 돌아온다.

실행 중에는 각 단계에서 환경으로부터 "실제 상황"을 파악하는 게 중요하다. 도구 호출 결과나 코드 실행 결과처럼 진행 상황을 확인하는 방법들이 있다. 에이전트는 체크포인트나 막히는 지점에서 사람의 피드백을 기다릴 수 있다. 작업은 보통 완료되면 끝나지만, 제어를 유지하기 위해 최대 반복 횟수 같은 종료 조건을 두는 것도 흔하다.

에이전트는 정교한 작업을 처리하지만 구현 자체는 대개 단순하다. 환경 피드백에 따라 도구를 쓰며 루프를 도는 LLM이 전부인 경우가 많다. 그래서 도구 세트와 문서를 명확하고 꼼꼼하게 설계하는 게 중요하다. 도구 개발 모범 사례는 부록 2("도구를 위한 프롬프트 엔지니어링")에서 자세히 다룬다.

**언제 쓰나:** 필요한 단계 수를 예측하기 어렵고 고정된 경로를 하드코딩할 수 없는 개방형 문제에 적합하다. LLM이 여러 차례 동작하게 되므로, 그 판단을 어느 정도 신뢰할 수 있어야 한다. 에이전트의 자율성은 신뢰할 수 있는 환경에서 작업을 확장할 때 강점이 된다.

자율성이 높다는 건 비용도 높고 오류가 쌓일 수 있다는 뜻이기도 하다. 샌드박스 환경에서 충분히 테스트하고 적절한 가드레일을 갖추길 권한다.

**예시:**
- SWE-bench 작업을 해결하는 코딩 에이전트 — 태스크 설명만 보고 여러 파일을 수정한다
- Claude가 컴퓨터를 직접 써서 작업을 수행하는 "컴퓨터 사용" 레퍼런스 구현

## 패턴을 조합하고 변형하기

이 빌딩 블록들은 정답이 아니다. 개발자가 각자의 상황에 맞게 다듬고 조합할 수 있는 공통 패턴이다. LLM 기능 전반에서 그렇듯, 성공의 열쇠는 성능을 측정하고 구현을 계속 개선하는 것이다. 복잡도는 그게 결과를 확실히 개선할 때만 올린다.

---

## 정리

LLM 세계에서 성공이란 가장 정교한 시스템을 만드는 게 아니다. 내 필요에 맞는 시스템을 만드는 것이다. 단순한 프롬프트로 시작하고, 종합적인 평가로 다듬고, 단순한 방법으로 안 될 때만 멀티스텝 에이전틱 시스템을 추가한다.

에이전트를 구현할 때 우리가 지키려는 세 가지 원칙이 있다.

- 에이전트 설계를 단순하게 유지한다
- 에이전트의 계획 단계를 명시적으로 드러내 투명성을 높인다
- 도구 문서화와 테스트를 꼼꼼히 해서 에이전트-컴퓨터 인터페이스(ACI)를 잘 만든다

프레임워크로 빠르게 시작하는 건 괜찮다. 하지만 프로덕션으로 가는 과정에서는 추상화 계층을 걷어내고 기본 컴포넌트로 만드는 걸 두려워하지 말라. 이 원칙들을 지키면 강력하면서도 믿을 수 있고, 유지보수하기 쉽고, 사용자가 신뢰하는 에이전트를 만들 수 있다.

## 부록 1: 현장의 에이전트

고객들과 일하면서 AI 에이전트가 특히 가능성을 보이는 두 분야를 발견했다. 위에서 설명한 패턴이 실제 가치를 내는 방식을 잘 보여준다. 둘 다 대화와 행동이 함께 필요하고, 성공 기준이 명확하고, 피드백 루프를 만들 수 있고, 사람의 감독이 의미 있게 개입되는 작업이다.

### A. 고객 지원

고객 지원은 친숙한 챗봇 인터페이스에 도구 통합을 통한 확장 기능을 붙인다. 개방형 에이전트에 자연스럽게 잘 맞는다.

- 지원 대화는 자연스럽게 대화 흐름을 따르면서 외부 정보와 행동이 필요하다
- 고객 데이터, 주문 내역, 지식베이스 문서를 가져오는 도구를 연결할 수 있다
- 환불 처리, 티켓 업데이트 같은 행동을 프로그래밍 방식으로 처리할 수 있다
- 성공 여부를 사용자가 정의한 해결 기준으로 명확히 측정할 수 있다

실제로 해결된 건에만 요금을 받는 사용량 기반 가격 모델로 서비스하는 회사들이 있다. 자기 에이전트의 효과를 그만큼 확신한다는 뜻이다.

### B. 코딩 에이전트

소프트웨어 개발 분야에서 LLM 기능은 코드 자동완성에서 자율 문제 해결로 빠르게 발전했다. 에이전트가 특히 잘 맞는 이유가 있다.

- 코드 솔루션은 자동화 테스트로 검증할 수 있다
- 에이전트가 테스트 결과를 피드백 삼아 솔루션을 반복 개선할 수 있다
- 문제 공간이 잘 정의되어 있고 구조화되어 있다
- 결과물 품질을 객관적으로 측정할 수 있다

우리 구현에서는 에이전트가 PR 설명만 보고 SWE-bench Verified 벤치마크의 실제 GitHub 이슈를 해결할 수 있다. 자동화 테스트로 기능 검증은 되지만, 솔루션이 더 넓은 시스템 요구사항에 맞는지는 사람 리뷰가 여전히 중요하다.

## 부록 2: 도구를 위한 프롬프트 엔지니어링

어떤 에이전틱 시스템을 만들든 도구는 핵심 구성 요소다. 도구는 Claude가 외부 서비스와 API와 상호작용하게 해준다. API에서 정확한 구조와 정의를 명시하면, Claude의 응답에 도구를 호출하려는 의도가 있을 때 응답에 도구 사용 블록이 포함된다. 도구 정의와 명세는 전체 프롬프트만큼 공들여 엔지니어링해야 한다.

같은 행동을 여러 방법으로 명세할 수 있다. 파일 수정을 예로 들면 diff로 표현하거나 파일 전체를 다시 쓸 수 있다. 구조화된 출력도 마크다운 안에 코드로 반환하거나 JSON 안에 넣을 수 있다. 소프트웨어 엔지니어링 관점에서 이런 차이는 외형적인 것이고 서로 무손실로 변환된다. 하지만 LLM이 쓰기 훨씬 어려운 형식이 있다. diff를 쓰려면 새 코드를 쓰기 전에 청크 헤더에서 바뀌는 줄 수를 알아야 한다. JSON 안에 코드를 넣으면(마크다운에 비해) 줄바꿈과 따옴표를 추가로 이스케이프해야 한다.

도구 형식을 결정할 때 우리가 쓰는 기준은 이렇다.

- 모델이 막힌 코너로 스스로를 몰기 전에 생각할 토큰을 충분히 준다
- 모델이 인터넷 텍스트에서 자연스럽게 접했을 형식과 가깝게 유지한다
- 수천 줄 코드의 줄 수를 정확히 세거나 작성한 코드를 문자열로 이스케이프하는 것처럼 형식상 "부담"이 없게 한다

경험적으로 이렇게 생각해본다. HCI(사람-컴퓨터 인터페이스)에 공을 들이는 만큼 ACI(에이전트-컴퓨터 인터페이스)에도 공을 들여라. 구체적으로는:

- 모델의 입장에서 보라. 설명과 파라미터만 보고 이 도구를 어떻게 쓰는지 바로 알 수 있는가? 아니라면 모델도 마찬가지다. 좋은 도구 정의에는 사용 예시, 엣지 케이스, 입력 형식 요구사항, 다른 도구와의 경계가 담겨 있다.
- 파라미터 이름이나 설명을 바꿔서 더 직관적으로 만들 수 있는가? 팀의 주니어 개발자를 위해 좋은 독스트링을 쓰는 것처럼 생각하라. 비슷한 도구가 많을수록 더 중요하다.
- 워크벤치에서 다양한 예시 입력으로 모델이 도구를 어떻게 쓰는지 확인하고, 실수하는 부분을 반복해서 개선하라.
- 실수하기 어렵게 도구를 만들어라(포카요케). 인자를 바꿔서 틀리기 어렵게 한다.

SWE-bench 에이전트를 만들 때 전체 프롬프트보다 도구 최적화에 더 많은 시간을 썼다. 예를 들어 에이전트가 루트 디렉터리를 벗어나면 상대 경로로 실수하는 걸 발견했다. 절대 경로를 항상 써야 하도록 도구를 바꿨더니 에이전트가 완벽하게 썼다.
