# 웹디자인 스킬 (레퍼런스 라이브러리 기반)

이 스킬은 "예쁘게"가 아니라 **검증된 디자인 레퍼런스 → 토큰 → 컴포넌트**로 일관되게 화면을 만든다.
`references/` 폴더에 여러 디자인 언어가 `design.md` 포맷(토큰+원칙)으로 들어 있고,
사이트 성격에 맞는 레퍼런스를 **골라서** 적용한다. 하나의 브랜드에 갇히지 않는다.

## 언제 쓰나
- 홈페이지/랜딩/섹션/컴포넌트를 디자인·구현할 때
- 새 고객사 사이트의 디자인 톤(테마)을 정할 때
- 여러 사이트를 관리하며 각 사이트에 맞는 일관된 디자인 언어가 필요할 때

---

## 레퍼런스 라이브러리 (`references/`)

각 파일은 하나의 완결된 디자인 언어다. `design.md` 포맷: 프론트매터에 토큰(colors·typography·
spacing·rounded·components), 본문에 특징·Do/Don't·반응형.

| 레퍼런스 | 성격 | 적합한 사이트 |
|---|---|---|
| `saas-stripe.md` | 인디고 CTA·그라디언트 메시·씬 타이포 | SaaS·핀테크·IT·마케팅 에이전시 |
| `medical-clinic.md` | 신뢰·클린·여백·차분한 블루/그린 | 내과·치과·정형외과·안과·한의원 |
| `editorial-luxury.md` | 세리프·큰 여백·절제된 고급감 | 성형외과·피부과·프리미엄 브랜드 |
| `minimal-swiss.md` | 타이포·그리드·무채색·모노 액센트 | 스튜디오·포트폴리오·B2B |
| `bold-brutalist.md` | 강한 대비·큰 글자·원색 | 에이전시·캠페인·이벤트·브랜드 |
| `friendly-rounded.md` | 둥근 모서리·파스텔·친근 | 소아과·산부인과·B2C·앱 랜딩 |

> `references/`는 시작점이다. 마음에 드는 사이트/스크린샷/피그마가 있으면 같은 포맷으로 파일을
> 추가하면 라이브러리가 커진다 (아래 "레퍼런스 추가하기" 참고).

## 워크플로우

### 1. 레퍼런스 선택 (Selection)
사이트 성격·업종·타깃·원하는 분위기를 파악해 위 표에서 **1개**를 고른다.
- 사용자가 특정 스타일/사이트를 지목하면 그걸 우선.
- 병원 분야는 진료과로 매핑: 성형·피부 → `editorial-luxury`, 소아·산부인 → `friendly-rounded`,
  그 외 대부분 → `medical-clinic`.
- 애매하면 후보 2개를 짧게 제시하고 고르게 한다.

### 2. 토큰 로드 (Tokenize)
선택한 레퍼런스 파일을 읽어 색·타이포·간격·라운드 토큰을 가져온다.
**모든 값은 토큰을 인용**한다 — 임의의 hex·px 남발 금지.

### 3. 브랜드 오버라이드 (Theme override, 멀티사이트 핵심)
고객사 고유 요소만 토큰 위에 덮어쓴다. 나머지(레이아웃·간격·컴포넌트)는 레퍼런스 그대로.
```
theme:
  brand-primary:  "#0071c2"   ← 고객사 메인 컬러로 교체
  brand-accent:   "#00b8a3"
  logo:           "/assets/client-logo.svg"
  font-display:   "Pretendard"   ← 한글은 Pretendard/Noto Sans KR 권장
```
→ 같은 레퍼런스로 사이트 A·B·C를 각기 다른 브랜드로 찍어낼 수 있다.

### 4. 구현 (Build)
`references/`의 컴포넌트 스펙 + 프로젝트의 실제 코드 패턴을 조합해 화면을 만든다.
- 정적 HTML/CSS가 기본 (프로젝트의 `venom-wordpress/preview`, `lib/post-designer.js`와 동일 방식, 의존성 0)
- React/Tailwind가 필요하면 토큰을 Tailwind config/CSS 변수로 매핑.

### 5. 검증 (Guardrails)
아래 공통 원칙 + 선택한 레퍼런스의 Do/Don't를 지켰는지 확인. 접근성·반응형 필수.

## 공통 디자인 원칙 (레퍼런스 무관, 항상 준수)

- **토큰 우선**: 색·폰트·간격·라운드는 토큰으로만. 하드코딩 값 최소화.
- **타입 스케일**: 한 페이지에 크기 남발 금지. 디스플레이/헤딩/본문/캡션 4~6단계로 제한.
- **여백이 디자인**: 섹션 간격은 넉넉히(데스크톱 64~96px). 요소를 붙이지 말 것.
- **CTA 위계**: 채움 버튼(primary)은 화면/밴드당 1개. 나머지는 아웃라인/텍스트.
- **한글 최적화**: 본문 line-height 1.6~1.75, 자간 -0.01~0em, Pretendard/Noto Sans KR.
- **접근성(A11y)**: 본문 대비 ≥ 4.5:1, 터치 타깃 ≥ 44×44px, 포커스 링 유지, 이미지 alt.
- **반응형**: 모바일 우선. 데스크톱→태블릿→모바일에서 그리드·타입이 단계적으로 축소.
- **일관성 > 화려함**: 같은 역할엔 같은 컴포넌트. 페이지마다 새 스타일 만들지 말 것.
- **성능**: 이미지 WebP·지연로딩, 렌더 차단 최소화 (SEO 속도 점수와 직결 → `seo-audit` 스킬).

## 레퍼런스 추가하기 (라이브러리 확장)

새 스타일을 넣으려면 `references/<name>.md`를 만들고 이 골격을 채운다:
```markdown
---
name: <스타일 이름>
description: <한 줄 특징>
colors: { primary, ink, canvas, canvas-soft, hairline, accent, ... }
typography: { display-xl, display-lg, heading, body, caption ... (size/weight/lineHeight/letterSpacing) }
rounded: { sm, md, lg, pill }
spacing: { xs, sm, md, lg, xl, xxl }
components: { button-primary, card, nav, input, ... }
---
## 특징 / ## 컬러 / ## 타이포 / ## 컴포넌트 / ## Do & Don't / ## 반응형
```
재료 확보 방법:
- **스크린샷/피그마**: 이미지를 주면 색·폰트·간격을 추출해 토큰화.
- **참고 사이트 URL**: 외부 접속(egress) 정책이 허용되면 HTML/CSS를 분석해 토큰화.
- **직접 지정**: "메인 컬러 #xxx, 세리프, 여백 크게" 식으로 값을 주면 그대로 작성.

## 짝 스킬 연동 (관리 콘솔 파이프라인)
- `ai-content-writer` → 글 생성 → **web-design**으로 브랜드 렌더 → `seo-audit`로 검증. 한 사이클.
- 사이트별 테마 토큰을 관리 콘솔 DB에 저장하면, 새 페이지 생성 시 자동으로 그 사이트 디자인이 입혀진다.
