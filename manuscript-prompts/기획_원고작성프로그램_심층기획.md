# 원고작성 프로그램 — 브레인스토밍 & 심층기획서 🛠️✍️

> 기준 문서: [`master_올인원_프롬프트.md`](./master_올인원_프롬프트.md)
> 한 줄 정의: **"분야·핵심·타겟만 넣으면, 다정한 톤의 헬스케어 원고와 수치 인포그래픽까지 한 번에 뽑아주는 단계형(위저드) 원고 생성 프로그램."**
>
> ※ 이 문서는 *글 한 편*이 아니라 **프로그램(앱) 개발**을 위한 기획서입니다.
> 올인원 프롬프트의 6단계를 그대로 소프트웨어의 화면·엔진·API로 옮깁니다.

---

## 0. 현황 진단 — 우리는 이미 절반쯤 와 있다

이 저장소엔 이미 **원고 자동생성 파이프라인**이 돌고 있어요. 맨땅에서 시작하는 게 아닙니다.

| 이미 있는 것 (`lib/`) | 하는 일 | 새 프로그램에서의 역할 |
| :--- | :--- | :--- |
| `post-generator.js` `generatePost()` | 키워드→LLM→검증→디자인 전 과정 오케스트레이션 | **재구성 대상** — 6단계 위저드로 분해 |
| `openai-client.js` `chatComplete()` | LLM 호출(현재 gpt-4o-mini) | **확장** — 모델 교체 가능하게 |
| `keyword-research.js` | 네이버·구글 연관어/질문 수집 | **브레인스토밍 엔진**의 입력원 |
| `medical-ad-validator.js` | 의료광고 금지어 검출·자동수정 | **가드레일** 그대로 재사용 |
| `content-validator.js` | 깨진 문자·빈 태그 정리 | **검수 단계** 재사용 |
| `infographics.js` (10종 SVG 렌더러) | 막대/링/카드/게이지 등 차트 SVG | **수치→인포그래픽 엔진**의 핵심 |
| `image-generator.js` `callDallE()` | AI 이미지 생성·GitHub 저장 | **이미지 인포그래픽 옵션** |
| `post-designer.js` | 브랜드 인라인 스타일 적용 | 출력 디자인 그대로 재사용 |
| `github-store.js` | 결과물 저장 | 초안/프로젝트 영속화 |

> **결론:** 새로 만들 건 "엔진"보다 **흐름(워크플로우)과 타겟·톤 레이어**입니다. 기존 자산의 70%를 재사용합니다.

### 지금 파이프라인의 한계 (= 우리가 메울 곳)

1. 입력이 `{category, keyword}`뿐 — **"누구를 위한 글인지(타겟)"가 없다.**
2. 톤이 SEO 정보글 위주 — **다정한 구어체 + 문어체 톤 스펙이 없다.**
3. 한 번에 통째로 생성 — **브레인스토밍·기획 단계가 없다** (사람이 고를 틈이 없음).
4. 인포그래픽이 자동 회전 선택 — 본문의 **수치와 정확히 매칭되지 않을 수 있다.**

## 1. 브레인스토밍 — 무엇을 만들 것인가 (아이디어 발산)

### 1-1. 핵심 문제

> "병원 마케터가 *타겟에 꼭 맞는*, *겁주지 않고 다정한*, *수치가 그림으로 보이는* 원고를
> 기획 단계부터 빠르게 만들고 싶다. 그런데 매번 프롬프트를 손으로 조립하는 건 비효율적이다."

### 1-2. 제품 아이디어 발산 (기능 후보 풀)

- 🧠 **주제 브레인스토밍 보드** — 키워드 넣으면 검색질문 15개 → 주제 후보 카드로
- 🎯 **타겟 스위처** — 직장인/환자/보호자 토글 한 번으로 같은 주제를 변주
- 🗣️ **톤 슬라이더** — "다정함 ↔ 전문적" 비율을 조절 (구어체/문어체 믹스)
- 📊 **수치 자동 시각화** — 본문 `[📊 그래프]` 마커를 인포그래픽으로 자동 변환
- 🛡️ **의료광고 안전등** — 실시간 금지표현 경고 (신호등 UI)
- 🔁 **시리즈 모드** — 1주제 → 5부작(타겟별)으로 확장
- 💾 **초안 저장/버전** — 단계별 결과를 프로젝트로 저장, 되돌리기
- 📤 **원클릭 발행** — 기존 `publish-post` 연계

### 1-3. 차별점 3가지 (이게 이 프로그램의 정체성)

1. **타겟 우선(Target-first)** — 글이 아니라 "누구"부터 정하고 모든 출력이 거기에 정렬된다.
2. **이중 톤 엔진** — 한 글 안에서 *공감은 구어체, 정보는 문어체*를 규칙으로 강제한다.
3. **수치-그림 1:1 매칭** — 지어낸 통계 차단 + 본문 수치와 인포그래픽을 정확히 연결한다.

## 2. 사용자 & 시나리오

### 2-1. 프로그램 사용자(운영자) — "마케터 지은"
- 병원 마케팅 담당, 디자이너 아님. "예쁜 그래프"를 코드 없이 원함.
- 하루 1~2편을 빠르게, 단 *심의에 안 걸리게* 내보내야 함.

### 2-2. 콘텐츠 타겟 = **프로그램의 1급 설정값** (사용자가 직접 설정)
> 사용자가 "타겟 설정"에서 고르거나 직접 정의 → 이후 모든 단계가 이 값에 종속.

```
TargetPersona {
  type: '직장인' | '환자본인' | '보호자' | '예방관심' | 'custom',
  detail: string,        // "하루 8시간 앉아 일하는 30대 직장인"
  condition?: string,    // 질환명 (환자/보호자일 때)
  pains: string[],       // 불안 3가지 (자동 생성)
  questions: string[],   // 궁금증 3가지 (자동 생성)
  desiredAction: string  // 유도할 행동 1가지
}
```

## 3. 제품 구조 — 올인원 6단계를 그대로 "위저드"로

올인원 프롬프트의 진행 순서(①~④) + 검수를 **6개 화면**으로 매핑합니다.

```
[Step 0] 입력      분야·핵심·채널·분량·브랜드색  ──┐
[Step 1] 브레인스토밍 검색질문→주제 후보 카드 (택1)   │  ← 사람이 고름(HITL)
[Step 2] 타겟 설정   페르소나 생성·확정 + 제목 3안   │
[Step 3] 원고 생성   이중 톤으로 본문 + [📊]마커     │  ← 엔진
[Step 4] 시각화      마커→인포그래픽/이미지 + 캡션    │
[Step 5] 검수·발행   체크리스트 점수 → 발행          ──┘
```

- **HITL(Human-in-the-loop):** Step 1·2는 반드시 사람이 확인·선택. (올인원의 "내가 고를게/다음" 지점)
- 각 Step 결과는 자동 저장 → 언제든 뒤로 가서 다시.

## 4. 핵심 엔진 4종 — 심층 설계

### (A) 브레인스토밍 엔진 `lib/brainstorm.js` (신규)
- 입력: `{field, core, channel}` + `researchKeywords()`(기존)로 실검색 질문 수집
- LLM 작업: 자연어 질문 15개 → 3분류(증상불안/정보비교/행동유도) → 주제 후보 5개(가제·추천타겟·수치아이디어)
- 출력: `TopicCandidate[]` (카드로 렌더)
- 재사용: `keyword-research.js`, `openai-client.js`

### (B) 타겟 페르소나 엔진 `lib/persona-engine.js` (신규)
- 입력: 사용자가 고른 `target.type` + 주제
- LLM 작업: 페르소나(불안3·궁금증3·행동1) + 타겟 맞춤 제목 3안 + 질문형 목차(`[📊]` 위치 표시)
- 출력: `TargetPersona` + `Outline`
- 변주 기능: 같은 주제로 `type`만 바꿔 재호출 → 비교 뷰

### (C) 이중 톤 엔진 `lib/tone-engine.js` (신규 — 가장 중요)
올인원의 **[말투 규칙]**을 시스템 프롬프트로 고정. 기존 `SYSTEM_PROMPT`의 *헬스케어 버전*.

```
규칙 요약(시스템 프롬프트로 주입):
- 공감/말걸기 = 구어체("~하시죠?","~거든요")  | 정보/통계/치료 = 문어체("~입니다")
- 어려운 용어는 괄호로 쉽게 풀기  | 모호한 수식어 대신 구체 수치
- 한 문단 3~4줄  | 과장·효능보장·공포마케팅 금지
- 수치 자리에는 [📊 그래프: (데이터)] 마커만 남길 것
- 소제목 질문형 + 즉답형 답변 먼저  | 끝에 FAQ 5개
```
- 페르소나·목차를 컨텍스트로 받아 `chatComplete()` 호출
- 직후 `validateMedicalAd()`+`autoFix()`(기존) → `reviewContent()`(기존)로 안전화

### (D) 수치 → 인포그래픽 엔진 `lib/chart-mapper.js` (신규, 기존 `infographics.js` 위)
- 본문에서 `[📊 그래프: ...]` 마커를 **정규식으로 추출**
- 각 마커의 데이터·유형을 LLM이 구조화 → `infographics.js`의 10종 렌더러 중 매칭
  (`horizontalBars`/`progressRings`/`statCards`/`gaugeList`/`stepFlow` 등)
- **출처 가드:** 출처 없는 수치는 `[출처 필요]` 플래그 → 사용자 확인 전 발행 차단 (환각 통계 방지)
- 옵션: "사진형 인포그래픽"이 필요하면 `image-generator.js`의 `callDallE()`로 AI 이미지 생성
- 출력: 마커 자리에 SVG/이미지 + 캡션(구어체+문어체) + `alt` 텍스트 삽입

## 5. 기술 아키텍처

### 5-1. 스택 (기존과 동일하게 — Vercel 서버리스)
- 프론트: 정적 HTML/JS 위저드 (`studio.html`) — 기존 `admin.html` 패턴 답습
- 백엔드: `api/*.js` 서버리스 함수 (Node ≥18, 기존 구조)
- 저장: `github-store.js`로 프로젝트/초안 영속화

### 5-2. API 설계 (단계별 1엔드포인트 — 상태는 클라이언트 보관)

| 엔드포인트 | 입력 | 출력 | 재사용 모듈 |
| :--- | :--- | :--- | :--- |
| `POST /api/brainstorm` | field, core, channel | TopicCandidate[] | brainstorm, keyword-research |
| `POST /api/persona` | topic, targetType | TargetPersona, titles[], outline | persona-engine |
| `POST /api/manuscript` | persona, outline, length | html(+[📊]마커), faq | tone-engine, medical-ad-validator |
| `POST /api/visualize` | html | html(+차트), captions[], warnings[] | chart-mapper, infographics, image-generator |
| `POST /api/review` | html | score, checklist[], fixes[] | content-validator, medical-ad-validator |
| `POST /api/publish` | finalHtml, meta | 발행 결과 | (기존) publish-post |

> `vercel.json`의 `functions`에 각 신규 함수 `maxDuration` 등록 필요(생성 계열 120s).

### 5-3. 데이터 모델 (프로젝트 = 단계의 누적)
```
ManuscriptProject {
  id, createdAt,
  input:    { field, core, channel, length, brandColor },
  topic:    TopicCandidate,          // Step1 확정
  persona:  TargetPersona,           // Step2 확정
  outline:  Section[],               // [📊] 위치 포함
  draft:    { html, faq },           // Step3
  visuals:  Chart[],                 // Step4 (data, type, source, caption, alt)
  review:   { score, checklist },    // Step5
  status:   'draft'|'reviewed'|'published'
}
```

### 5-4. LLM 오케스트레이션 & 모델 선택
- 현재 `openai-client.js`는 OpenAI 고정(`gpt-4o-mini`). → **모델 어댑터로 분리**해 교체 가능하게.
- 권장: 단계별로 모델 차등(비용·품질 균형)
  - 브레인스토밍·검수 = 저비용·빠른 모델
  - 원고 생성(톤이 핵심) = 고품질 모델
- Claude로 갈 경우 모델 ID: `claude-opus-4-8`(최고품질) / `claude-sonnet-4-6`(균형) / `claude-haiku-4-5-20251001`(저비용).
  *다정한 구어체+문어체 믹스는 글쓰기 품질 의존도가 높아 생성 단계는 상위 모델 권장.*
  (기존 OpenAI 경로는 유지하되 `LLM_PROVIDER` 환경변수로 선택 — 점진 이전)

## 6. 엔드투엔드 데이터 흐름

```
사용자 입력(분야·핵심·타겟)
   │
   ▼  /api/brainstorm ── keyword-research(네이버·구글) + LLM
주제 후보 카드 ──(사람이 택1)
   │
   ▼  /api/persona ── LLM
페르소나 + 제목3안 + 목차[📊]
   │
   ▼  /api/manuscript ── tone-engine(이중톤) + 의료광고 가드
본문 HTML + [📊 마커] + FAQ
   │
   ▼  /api/visualize ── chart-mapper → infographics(SVG)/DALL·E
차트 삽입 + 캡션 + 출처경고
   │
   ▼  /api/review ── 체크리스트 점수
발행 가능 → /api/publish
```

## 7. MVP 범위 & 로드맵

### 🟢 Phase 0 — 톤 PoC (가장 먼저, 1~2일)
- `tone-engine.js`만 구현 → 기존 `generatePost`에 "타겟+이중톤" 옵션으로 끼워 검증

### 🔵 Phase 3 — 확장
- 시리즈 모드, 영어 번역(기존 `translate.js`), 발행 자동화, 프로젝트 버전관리
