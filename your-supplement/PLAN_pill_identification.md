# 💊 약 검색·사진 식별 기능 — 분석 기획서

> 작성: 2026-07-01 · 대상: 당신의영양제(당신의약국) · 목적: "약 검색(모양·구성) + 사진/카메라 식별" 기능의 서비스 방향·배치·데이터·기술·로드맵을 정밀 설계.
> 전제: 우리는 **질병 진단·처방이 아닌 "근거 기반 정보 제공" 서비스**. 이 기능도 같은 원칙(정보 제공·최종 확인은 약사/의사)을 따른다.

---

## 1. 배경 & 왜 지금인가

### 1-1. 사용자 페인포인트 (진짜 문제)
- **"이 알약 뭐지?"** — 약통에서 빠진 낱알, 부모님 약봉지, 여행 중 섞인 약, 유통기한 지난 약 서랍 → **정체 불명 알약**을 확인할 방법이 마땅치 않음.
- **"내가 먹는 약 + 영양제 같이 먹어도 되나?"** — 우리 앱의 핵심(상호작용·DUR)과 직결. 근데 **사용자가 약 이름을 정확히 모름**(포장 버림, 성분명 어려움).
- 기존 검색은 **약 이름/성분을 알아야** 가능 → 진입장벽. **모양·색·각인·사진**으로 역추적하는 경로가 필요.

### 1-2. 우리 서비스와의 시너지 (차별점)
- 경쟁 낱알식별 서비스(약학정보원·굿닥 등)는 "이게 무슨 약"까지만. 우리는 **식별 → 그 약 ↔ 영양제 상호작용·DUR·복용 스케줄**로 자연스럽게 연결 → **"약을 식별하면 내 영양제 조합까지 점검"** = 유일한 흐름.
- 우리의 `/analyze`(조합분석)·DUR·`interactions.json`이 이미 있음 → 식별 결과를 **바로 물려줄 파이프라인 보유**.

## 2. 시장·경쟁 분석

| 서비스 | 약 검색 | 낱알(모양) 검색 | 사진/카메라 식별 | 영양제 상호작용 연결 |
|---|---|---|---|---|
| 약학정보원(health.kr) | ✅ | ✅(모양·색·각인 필터) | ✕ | ✕ |
| 식약처 의약품안전나라(nedrug) | ✅ | ✅ 낱알식별 | ✕ | 부분(DUR) |
| 굿닥·똑닥 등 | ✅ | 일부 | 일부(제한적) | ✕ |
| 필리(Pilly)/필리케어 | 영양제 위주 | ✕ | ✕ | ✅(영양제만) |
| **당신의영양제(우리)** | (신규) | (신규) | **(신규·차별)** | **✅ 이미 보유** |

**결론:** "낱알 필터 검색"은 흔함(패리티 필요). **"사진 식별 + 영양제 상호작용 연결"** 조합은 비어 있는 자리 → 우리의 진입 지점.

## 3. 기능 정의 (2트랙)

### 트랙 A — 약 검색 (텍스트·필터) · **MVP 필수**
1. **이름/성분 검색**: "타이레놀", "아세트아미노펜" → 제품 정보(효능·용법·주의·성상).
2. **낱알 필터 검색**: **모양**(원형·타원·장방형…) + **색**(흰색·노랑…) + **각인/식별문자**(예: "TYLENOL", "APAP") + **제형**(정제·캡슐) → 후보 목록 + 이미지.
3. 결과 카드: 제품명·성분·효능·**낱알 이미지**·업체 → 상세(e약은요) + **"내 영양제와 궁합 보기"(→ /analyze 연동)**.

### 트랙 B — 사진/카메라 식별 · **차별 기능 (2단계)**
> ⚠️ **핵심 설계 원칙**: 현장 인식률 한계(연구상 시범 ~15%) 때문에 **"자동 확진"이 아니라 "후보 좁혀주는 보조 도구"**로 포지셔닝. 자동 인식 결과 + **사용자 확인 필터**를 결합.

**플로우:**
1. 사진 촬영/업로드 (알약 클로즈업).
2. **OCR로 각인 텍스트 추출** (네이버 클로바 OCR / 구글 비전) + **색/모양 자동 추정**(보조).
3. 추출값으로 **식약처 낱알식별 DB 자동 조회** → **후보 N개** 제시(단정 X).
4. 사용자가 **색·모양·각인**을 눈으로 확인·보정(자동값 미리 채움) → 후보 압축.
5. 확정 후보 → 상세정보 + **DUR·영양제 상호작용 연결**.

**정직성 장치(필수):**
- "AI 자동 인식은 참고용입니다. **최종 확인은 반드시 약사·의사**." 상시 고지.
- 결과에 **신뢰도/후보 순위** 표기, 단일 확정 강제 금지.
- 촬영 가이드(밝은 곳·각인 정면·1알) 제공해 실패율↓.

## 4. 데이터 소스 (어디서 가져오나)

| 데이터 | 소스 | 용도 | 키/신청 |
|---|---|---|---|
| **낱알식별**(모양·색·각인·이미지·제품명·업체) | 식약처 **의약품 낱알식별 정보** OpenAPI (data.go.kr `15057639`) | 트랙 A 필터검색 · 트랙 B 매칭 DB | `DATA_GO_KR_KEY`(활용신청) |
| **약 개요**(효능·용법·주의·저장) | 식약처 **e약은요**(의약품개요정보, `15075057`) | 검색·식별 결과 상세 | 동일 키 |
| **병용금기/DUR** | 식약처 DUR(`15059486`) | 식별 약 ↔ 영양제 안전 경고 | 동일 키 |
| **각인 텍스트 OCR** | 네이버 CLOVA OCR **또는** Google Cloud Vision | 사진→각인 문자 추출 | 유료 API 키 |
| (선택) 색/모양 분류 | 자체 경량 모델 or 규칙기반(HSV 색·윤곽 모양) | 자동 필터 프리필 | 자체 |

> 우리 `DATA_GO_KR_KEY`(약국·응급실에 이미 쓰는 계정)에 **낱알식별·e약은요·DUR 데이터셋 활용신청만 추가**하면 됨. OCR만 별도 상용 키.
> 대량 낱알 이미지·메타는 **배치로 `data/`에 캐시**(검색 속도·비용 절감), 상세는 실시간.

## 5. 사이트 내 배치 (IA · 정보구조)

### 5-1. 신규 상위 메뉴: **"약 알기"(약 검색·식별)**
현재 nav: `조합분석 · 제품찾기 · 랭킹 · 성분사전 · 내 루틴 · 약국`
→ **`💊 약 알기`** 추가 (또는 `제품찾기` 옆). 하위:
- **약 검색**(`/drug` 또는 `/medicine`) — 이름·성분·낱알 필터
- **사진으로 찾기**(`/drug/scan`) — 카메라 식별 (트랙 B)

### 5-2. 진입점 다중화 (발견성)
- **홈 히어로 하단** "이 약 뭐지? 📷 사진으로 찾기" CTA.
- **/analyze(조합분석)** 안에 "약 이름 몰라요? → 사진으로 찾기" 링크 → 식별 후 자동으로 조합분석에 주입.
- **DUR/복용약 선택** 지점에 "사진으로 약 추가" 옵션.

### 5-3. 결과 → 액션 연결(핵심 흐름)
```
[사진/검색] → [약 식별] → [상세정보]
                         ├─ 내 영양제와 궁합 보기 → /analyze 자동주입
                         ├─ 병용금기(DUR) 경고
                         └─ 내 약 목록에 저장(내 루틴 연동)
```

## 6. 기술 아키텍처

```
[클라이언트] 카메라/업로드
    │ (이미지)
    ▼
[/api/drug/ocr] ── CLOVA/Vision OCR ──► 각인 텍스트 + (색·모양 추정)
    │
    ▼
[/api/drug/identify?imprint=&color=&shape=] ── 낱알식별 DB(캐시/실시간) ──► 후보 목록
    │
    ▼
[결과 UI] 후보 카드 → e약은요 상세 + DUR + /analyze 주입
```

- **라우트(신규)**: `/api/drug/search`(텍스트·필터), `/api/drug/ocr`(이미지→각인), `/api/drug/identify`(매칭), `/api/drug/detail`(e약은요).
- **config-driven**: OCR·data.go.kr URL/키는 env 주입, 미설정 시 graceful(샘플·"준비중"). 기존 패턴 준수.
- **개인정보**: 업로드 사진은 **서버 미저장(휘발 처리)** 또는 즉시 폐기 원칙 명시.

## 7. 단계별 로드맵

| 단계 | 범위 | 산출물 | 의존 |
|---|---|---|---|
| **P0 (MVP-A)** | 약 **텍스트·낱알 필터 검색** + 상세 + /analyze 연동 | `/drug` 페이지, `/api/drug/*`(낱알식별·e약은요) | `DATA_GO_KR_KEY` 활용신청 |
| **P1 (MVP-B)** | **사진 식별**(OCR 각인 → 후보) + 사용자 보정 필터 | `/drug/scan`, `/api/drug/ocr` | OCR 키(클로바/비전) |
| **P2** | 색·모양 **자동 프리필**(규칙기반), 촬영 가이드, 신뢰도 표기 | 정확도·UX 개선 | 자체 |
| **P3** | 식별 약 **내 루틴 저장**·DUR 상시 경고·재구매/복약 알림 연동 | 리텐션 | 카카오 알림톡(기존 계획) |

> **먼저 P0(검색)**: OCR 없이도 즉시 가치. **P1(사진)**은 차별 기능이나 OCR 비용·정확도 검증 후.

## 8. 리스크 & 대응

| 리스크 | 영향 | 대응 |
|---|---|---|
| **사진 인식률 낮음**(현장 ~15%) | 오식별→신뢰 하락 | "보조 도구"로 포지셔닝 · 후보 다중 제시 · 사용자 확인 필수 · 촬영 가이드 |
| **오식별로 인한 복약 사고** | 안전·법적 | 상시 면책("최종 확인 약사") · 단일 확진 금지 · DUR 경고 우선 |
| **의료기기/광고 심의** | 규제 | "정보 제공"만, "진단·처방" 표현 배제 · 식약처 공개데이터 근거 명시 |
| **OCR API 비용** | 운영비 | P0(검색) 우선 · 사진은 호출 최적화(전처리·캐시) · 무료쿼터 활용 |

| **낱알 이미지 저작권/이용범위** | 법적 | 식약처 공개데이터 이용조건 확인·출처표기 |

## 9. 성공지표 (KPI)
- **활성화**: 사진 식별 시도 수 / 검색 수, 식별→상세 전환율.
- **핵심 전환(차별)**: 식별/검색 → **/analyze(조합분석) 진입률** (우리만의 가치).
- **정확도(체감)**: 후보 상위 3개 내 정답 포함률(사용자 확인 기반).
- **안전**: 면책·약사확인 노출률(상시 100%).
- **리텐션**: 식별 약 "내 약 저장" → 복약/상호작용 재방문.

## 10. 결론 & 권고

1. **방향**: "이 약 뭐지?"를 **사진/모양으로 역추적** → 우리 강점인 **영양제 상호작용·근거 정보로 연결**. (경쟁사엔 없는 흐름)
2. **정직 설계**: 사진 식별은 **자동 확진이 아닌 후보 보조**. 최종 확인은 약사 — 이 원칙이 신뢰·법적 안전의 핵심.
3. **실행 순서**: **P0(텍스트·낱알 검색, 식약처 공개데이터, 키 활용신청만)** 부터 → 즉시 가치. 이어 **P1(사진 OCR 식별)** 로 차별화.
4. **데이터**: 기존 `DATA_GO_KR_KEY`에 **낱알식별·e약은요·DUR 활용신청 추가** + **OCR 상용키(사진 단계)**.

> 다음 액션 제안: **P0 상세 설계·와이어프레임 → `/drug` 검색 스캐폴드(config-driven) 구현**. 승인 시 진행.
