# 🌐 당신의영양제 — 글로벌 데이터 전략 (시판 모든 영양제 + 세계 권위 출처)

> 작성: 2026-06-29 · 목표: "현재 시판 중인 모든 영양제"를 다루고, 효능·안전 근거를 **세계 권위 기구**로 뒷받침.

---

## 0. 핵심 구조 결정 (가장 중요)

**"시판 모든 영양제(제품)"와 "효능·안전(성분)"은 다른 레이어다.**

```
[제품 레이어]  시판 제품 수만~수십만 종  ── 제품 라벨 DB(공식 API)
       │  (제품 → 성분·함량 매핑)
       ▼
[성분 지식베이스(KB)]  근거·용량·상호작용  ── 세계 권위 기구
       │  (추천 엔진은 여기서 추론)
       ▼
[추천/안전/루틴]  현재 완성된 웹앱
```

- 제품은 결국 **성분의 조합(제형)**. 추천·안전·근거는 **성분 KB**가 책임지고, 제품은 "이 성분이 든 실제 제품"을 **검색·구매·라벨대조**하는 레이어로 붙인다.
- 따라서 "모든 영양제"는 ① **제품 라벨 DB(공식 API)** + ② **성분 KB를 포괄적으로 확장**으로 달성한다. 제품을 일일이 크롤링할 필요가 없다.

---

## 1. 현실 점검 — 크롤링 vs 공식 API

- 제약사 사이트 **대량 크롤링은 비권장**(약관·차단·법적 리스크·품질 들쭉날쭉). **이미 정부가 라벨 데이터를 공식 API로 개방**하고 있어 그걸 쓰는 게 정답.
- **이 개발 샌드박스는 외부망 차단**이라 여기서 직접 호출은 불가하지만, **배포(Vercel) 환경은 네트워크 개방** → API 연동 코드는 **배포 시 실제 동작**한다(현재 offers·nearby가 그 방식).

## 3. 단계 로드맵

### G1. 성분 KB 스키마를 '확장 가능' 구조로 정비
- 현재 17종 → 수백 종으로 늘 수 있게 `ingredients`에 `aliases`(동의어·영문·원료명), `synonyms_for_matching`(라벨 매칭용) 추가.
- 근거는 `evidence.json`에 출처별(`ODS/NCCIH/WHO/EFSA/Cochrane`) 누적.

### G2. 제품 검색 API 연동 (배포에서 동작)
- `/api/products?q=` → **식약처 건기식 품목 API**(국내) + **DSLD**(해외) 호출, 라벨 성분을 KB id로 매핑, 키 없으면 샘플.
- 결과 화면/검색에서 "이 성분이 든 실제 제품" 노출 → 추천 성분과 연결.

### G3. 성분 KB 포괄 확장 (권위 출처 큐레이션)
- 식약처 **기능성 원료 인정현황 API**로 국내 고시·개별인정 원료를 **자동 적재**(D1·D2 동시 해결).
- DSLD/ODS로 해외 주요 성분 보강. 각 항목 출처·근거강도 인용.

### G4. 안전(DUR)·이상사례 보강
- openFDA CAERS로 성분별 이상사례 신호, 식약처 DUR로 병용금기 확장(P5 연속).

### G5. 거버넌스
- 각 데이터에 `source`·`reviewed_at`·`api_synced_at`. 자동 동기화 + 검수 로그.

---

## 4. 한계·전제 (정직하게)

- **대량 크롤링은 안 한다**(불안정·리스크). 공식 API·공개 데이터셋만.
- 효능 근거는 `citable:true`만. 상업 DB(Pillyze·Labdoor)는 **참고/벤치마크**로만.
- 일부 권위문서(WHO·Codex·EMA·NCCIH)는 **API가 없어 PDF 인용** → 자동적재 불가, 큐레이션 필요.
- 키 필요: 식약처 품목/원료 API(data.go.kr 활용신청), (DSLD는 키 불필요).

---

## 5. 추천 시작점

**G2(제품 검색 API) + G3 일부(식약처 원료 인정 API 적재)** 를 함께. 이유:
- DSLD는 키 없이 즉시 붙고, 식약처 품목/원료 API는 이미 보유한 `DATA_GO_KR_KEY` 계열로 활용신청만 하면 됨.
- "시판 모든 영양제"라는 사용자 목표를 **제품 검색**으로 가시화하면서, 성분 KB를 **공식 데이터로 자동 확장**해 신뢰도(D1·D2)까지 동시 해결.
- 배포 환경에서 실제 동작(샌드박스 제약과 무관).
